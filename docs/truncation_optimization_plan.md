# OCR/LLM 截断优化计划

> 日期: 2026-06-05 | 状态: 待实施 | 优先级: P1

## 问题描述

当前系统存在 3 层截断链路，导致大文档（200+页PDF）的关键信息（死亡诊断、鉴定结论、手术记录等经常在文档后半段）被丢失：

```
原始PDF (200+页, 5万+字OCR)
    ↓ _build_structured_context()
第1层截断: 每份材料 OCR 原文 2000字(普通)/5000字(关键类别)
    ↓ 合并所有材料 → 可能 10万+字
第2层截断: _extract_document_slots() 取前 30000字
    ↓ 段落生成上下文
第3层截断: llm_extractor.py 每个槽位 8000字
    ↓ 发给 LLM
输出结构化数据 + 段落文本
```

### 实际影响案例

- 200页住院病历，OCR约5万字，前5000字只有入院记录和首次病程 → 死亡诊断在第180页被截断
- 10个材料×5000字 = 50000字，合并上下文30000字 → 后1/3材料的信息完全丢失
- 段落2（诊疗经过）需要完整时间线，但上下文只有8000字

### 已有的间接缓解（本次 Phase 1-3 修复）

- `_direct_extract_death_diagnosis()`: 绕过LLM，直接从完整OCR原文搜索 → 不受30000字限制
- `_direct_extract_appraisal_conclusion()`: 同理
- `_direct_extract_staff_qualification()`: 同理
- **但**: 如果OCR原文本身也被截断（第1层），定向提取也无法工作

## 优化方案（4项）

### 优化1: 智能上下文构建 — 按信息密度切片替代暴力截取

**现状**: `_build_structured_context()` 对每份材料的OCR原文取前N字
**目标**: 按段落重要性排序拼接，确保关键信息不丢失

**实现思路**:
1. 将每份材料的OCR原文按段落（`\n\n`或换行+缩进）拆分为片段
2. 给每个片段打重要性标签：
   - 🔴 关键: 含"死亡诊断/鉴定意见/手术记录/尸检/执业证/身份证"等关键词
   - 🟡 重要: 含"诊断/治疗/检查/出院/入院/医嘱/护理"等关键词
   - ⚪ 一般: 其他段落
3. 按优先级拼接：先放🔴关键段落 → 🟡重要段落 → ⚪一般段落（直到达到字符上限）
4. 同一材料内的段落保持原始顺序（仅调整不同优先级之间的顺序）

**影响文件**: `services/evidence/document_analyzer.py` → `_build_structured_context()`
**参数配置化**: `settings.context_max_per_material` (默认8000) / `settings.context_max_total` (默认80000)

### 优化2: LLM 调用分槽提取 — 大文档拆分多次调用

**现状**: `_extract_document_slots()` 一次性提取所有字段，30000字上限
**目标**: 按信息类别拆分为3-4次独立调用，每次只需关注少量上下文

**实现思路**:
```
调用1: 提取身份信息 (原告/被告/患者) — 只需身份证+户口本类材料
调用2: 提取诊疗信息 (入院/诊疗/转院) — 只需病历类材料
调用3: 提取鉴定信息 (鉴定结论/人员资质) — 只需鉴定+资质类材料
调用4: 提取费用信息 — 只需票据类材料
```

每次调用只传入相关分类的材料上下文，而非全部材料。这样：
- 每次调用上下文更小更聚焦（~10000字/次 vs 当前30000字一次性）
- 提取准确率更高（prompt更精确，上下文噪音更少）
- 不再需要全局30000字截断

**影响文件**: `services/evidence/document_analyzer.py` → `_extract_document_slots()`

### 优化3: 提升截断阈值 — 利用 qwen3.5-plus 128K context

**现状**: `_extract_document_slots()` 硬编码 `max_chars = 30000`（约7500 tokens），仅用qwen3.5-plus的6%
**目标**: 合理利用长文本模型能力

**实现思路**:
- `_extract_document_slots()`: `max_chars = 30000` → `80000`（约20000 tokens，占128K的16%）
- `_build_structured_context()` 关键类别: `5000` → `15000`字
- `_build_structured_context()` 普通类别: `2000` → `5000`字
- `llm_extractor.py` 槽位提取: `8000` → `15000`字
- 所有截断阈值通过 `settings.py` 配置化，不硬编码

**注意**: 更大的上下文 = 更高的 token 成本和更慢的响应。需要实测平衡点。
**建议**: 先提升到上述值，观察实际 token 用量和准确率，再调整。

**影响文件**: 
- `config/settings.py` — 新增配置项
- `services/evidence/document_analyzer.py` — 引用配置
- `services/complaint/llm_extractor.py` — 引用配置

### 优化4: OCR 输出完整性保障

**现状**: `evidence_materials.ocr_text` 存储的是 OCR 全文，但百炼OCR引擎本身有输出token限制
**目标**: 确保OCR原文100%完整存储到数据库

**实现思路**:
1. 检查百炼OCR模型（qwen-vl-ocr）的输出限制 — 如果单页输出被截断，需要分块识别
2. 大PDF（>50页）分批OCR：每批10-15页，避免单次API调用返回token超限
3. OCR结果合并时保留页码标记，方便后续定向提取定位
4. OCR完成后校验：如果某页OCR结果为空但PDF该页有文字，标记为需重试

**影响文件**: 
- `services/ocr/bailian_engine.py` — 分批识别逻辑
- `services/complaint/ocr_service.py` — 合并+页码标记
- `worker/evidence_tasks.py` — OCR管线调整

## 实施计划

| 序号 | 优化项 | 优先级 | 复杂度 | 预计改动量 |
|------|--------|--------|--------|-----------|
| 1 | 优化3: 提升截断阈值+配置化 | P0 | 低 | ~50行（改配置+引用） |
| 2 | 优化1: 智能上下文构建 | P1 | 中 | ~150行（重写_build_structured_context） |
| 3 | 优化2: 分槽提取 | P1 | 高 | ~300行（重写_extract_document_slots） |
| 4 | 优化4: OCR完整性保障 | P2 | 中 | ~100行（分批识别+校验） |

### 建议执行顺序

```
优化3 (快速见效) → 优化1 (核心提升) → 优化2 (架构优化) → 优化4 (兜底保障)
```

**优化3** 最快见效（10分钟改完），改完立即可部署验证。
**优化1** 是核心改动，需要仔细测试各种边界情况。
**优化2** 改动最大但效果最好，需要验证分槽提取的准确率。
**优化4** 是兜底措施，确保OCR源头不出问题。

## 风险评估

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 更大上下文导致LLM响应变慢 | 中 | 中 | 实测token耗时，必要时用flash模型 |
| 智能切片排序逻辑引入bug | 中 | 高 | 保留原始暴力截取作为fallback |
| 分槽提取字段遗漏 | 低 | 高 | 增加字段完整性校验 |
| 百炼OCR分批后合并格式不一致 | 低 | 中 | 增加格式标准化步骤 |
