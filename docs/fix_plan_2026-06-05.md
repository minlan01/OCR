#### 1.1 大 PDF 上传 + No space left on device + File storage failed

**问题根因**：
- 当前上传方式：`content = await file.read()` 把整个文件读入内存 → 500MB PDF 占 500MB 内存
- 10 人同时上传 → API 容器（1G 内存）直接 OOM
- Worker 容器（2G 内存）处理大 PDF 时，PyMuPDF 全量加载 → 内存不够 → /tmp 写临时文件 → 磁盘满
- MinIO 上传失败只有一次机会，没有重试

**修复方案**：

##### A. 流式上传改造（evidence.py）

**当前问题**：`content = await file.read()` 一次读全量 → 500MB 吃 500MB 内存

**改为**：流式分片上传到 MinIO

```python
# evidence.py upload_materials() 改造
MINIO_MULTIPART_THRESHOLD = 100 * 1024 * 1024  # 100MB 以上走分片
MINIO_PART_SIZE = 50 * 1024 * 1024              # 每片 50MB

async def upload_materials(...):
    for file in files:
        # 1. 先检查 Content-Length（不读内存）
        if file.size and file.size > settings.max_upload_size:
            raise HTTPException(400, detail=f"文件过大...")

        # 2. 小文件（<100MB）：保持原方式，一次 read + upload_bytes
        # 3. 大文件（≥100MB）：流式读取 + MinIO 分片上传
        if file.size and file.size >= MINIO_MULTIPART_THRESHOLD:
            # 使用 minio_client.streaming_upload()
            # 前端 chunk 读取，后端边读边传
            # 内存峰值 = 一个分片大小（50MB）
            minio_key = await _stream_upload_to_minio(file, case_id, EVIDENCE_MINIO_BUCKET)
            file_size_actual = ...  # 从 MinIO 获取
        else:
            content = await file.read()
            # ... 原逻辑
```

**MinIO 分片上传**（minio_client.py 新增方法）：
- `streaming_upload(bucket, object_key, file_like, content_type, part_size=50MB)`
- 使用 `minio.client.fput_object()` 或 `mpu = minio_client._initiate_multipart_upload()` + 逐片上传
- 上传完成后验证 ETag

**内存优化**：
- 上传阶段内存峰值：小文件 500MB（上限）→ 大文件 50MB（一块）
- API 容器不会因同时上传而 OOM

##### B. 容器临时文件管理（docker-compose.yml + evidence_tasks.py）

**当前问题**：worker 的 /tmp 和根磁盘共享，大 PDF 的图片撑爆磁盘

**改为**：tmpfs 仅放当前页图片 + MinIO 临时桶放中间产物

```yaml
# docker-compose.yml - worker 服务
worker:
  tmpfs:
    - /tmp:size=128M    # 仅放当前页图片（每页 <10MB）
  volumes:
    - scanstruct_templates:/app/template_output
```

**临时文件生命周期管理**（evidence_tasks.py）：
- 每次处理完一批页面（5-10 页）→ 立即清理 /tmp 下该批图片
- OCR 文本结果直接写 MinIO 临时桶 `scan-ocr`，不落本地
- 新增 `minio_temp_prefix = "temp/"` → 任务完成后统一清理
- 新增 `_cleanup_temp_files()` 函数，在每个 Celery 任务结束时调用

##### C. PDF 按需分页处理（evidence_tasks.py）

**当前问题**：PyMuPDF 一次性加载整个 PDF → 500MB PDF 内存可能 1.5G+

**改为**：按批次分页提取

```python
# evidence_tasks.py - OCR 任务改造
PDF_BATCH_PAGES = 10  # 每批处理 10 页

def _process_pdf_in_batches(minio_key: str, case_id: str) -> list[str]:
    """按批次处理 PDF，内存恒定"""
    # 1. 流式下载 PDF 到 /tmp（仅下载一次，tmpfs 128M 足够放一个临时 PDF）
    pdf_path = _download_from_minio_to_tmp(minio_key)

    # 2. 打开 PDF（PyMuPDF 惰性加载，不全部解压到内存）
    doc = fitz.open(pdf_path)
    total_pages = doc.page_count

    all_ocr_texts = []
    for start in range(0, total_pages, PDF_BATCH_PAGES):
        end = min(start + PDF_BATCH_PAGES, total_pages)
        # 3. 提取本批页面 → 图片 → OCR → 文本
        batch_texts = _ocr_page_batch(doc, start, end)
        all_ocr_texts.extend(batch_texts)

        # 4. 清理本批图片（/tmp 只保留当前批）
        _cleanup_page_images(start, end)

    # 5. 最终清理
    doc.close()
    os.unlink(pdf_path)

    return all_ocr_texts
```

**关键策略**：
- 文字型 PDF（文本页）：直接 `page.get_text()`，不走 OCR（极快，零额外内存）
- 扫描型 PDF：逐页提取图片 → 发 OCR → 立即删除图片
- 采样逻辑已有（`pdf_classifier.py`），加强对大文件的处理

##### D. MinIO 上传重试（minio_client.py）

- 上传失败自动重试 3 次，指数退避（1s → 2s → 4s）
- 封装为 `_upload_with_retry()` 私有方法
- 新增 `upload_streaming()` 方法（分片上传）
- 记录上传耗时和文件大小到日志

##### E. 上传限流调整（evidence.py）

- 当前 batch-upload 限流 `3/minute` → 改为 `10/minute`
- 大 PDF 上传耗时长，不应被限流卡住

##### F. 上传文件类型扩展（evidence.py + settings.py）

- `settings.allowed_extensions` 增加：`.mp3 .wav .m4a .amr .aac`（音频）
- `_detect_file_type()` 新增 `audio` 类型
- `ALLOWED_TYPES` 增加 `"audio"`
- 音频文件 OCR 状态标记为 `not_applicable`
- 音频 `file_type="audio"` → 分类时归入新类别 `audio_recording`
- 前端展示：音频文件图标 + 简易播放器（不识别内容）
- 分类 prompt 新增音频分类规则

**涉及文件**：
- `docker-compose.yml` — worker tmpfs 128M
- `services/storage/minio_client.py` — 流式上传 + 重试
- `api/routes/evidence.py` — 流式上传 + 限流 + 文件类型扩展
- `worker/evidence_tasks.py` — PDF 分批处理 + 临时文件清理
- `config/settings.py` — allowed_extensions 扩展 + 新增配置项
- `db/models_evidence.py` — file_type CHECK 约束增加 audio
- `services/evidence/classifier.py` — 新增 audio_recording 分类

#### 1.2 解决 HTTP 429 API 限流

**问题根因**：
- 10 人同时使用 → 大量 LLM API 调用（OCR + 分类 + 提取 + 段落生成）
- OCR 并发 25 + 文本分析并发 5 → 300 个用户可能同时 300 个请求
- DashScope API 限流 → 429 错误传播

**修复方案**：

##### A. 全局 LLM 并发信号量（新建 `services/llm/rate_limiter.py`）

使用 Redis 分布式信号量控制跨 worker 的 LLM 调用总数：

```python
class LLMRateLimiter:
    """基于 Redis 的分布式 LLM 调用限流器"""

    # 并发上限（可从 settings 读取）
    DEFAULT_LIMITS = {
        "text": 5,        # 文本分析最多同时 5 个
        "ocr": 10,        # OCR 最多同时 10 个
        "flash": 15,      # 快速模型（分类）最多 15 个
    }

    async def acquire(self, model_type: str, timeout: float = 60.0) -> bool:
        """获取调用许可，超时返回 False"""
        key = f"llm_semaphore:{model_type}"
        # Redis INCR + EXPIRE 实现简易信号量
        # 超时等待而非直接失败

    async def release(self, model_type: str):
        """释放调用许可"""
        key = f"llm_semaphore:{model_type}"
        # Redis DECR
```

##### B. 429 自动重试

检测 429 状态码 → 指数退避重试（最多 3 次）
- 重试间隔：2s → 4s → 8s
- 重试仍失败 → 任务标记为 `retrying` 状态，不直接 `failed`
- 重试时自动降低并发上限（5 → 3 → 1）

##### C. 降级策略

- **OCR 阶段**：429 时自动降低并发（25 → 10 → 5）
- **文本分析阶段**：429 时串行处理（逐个材料，不再并行）
- **bailian_ocr_max_concurrent** 从硬编码 25 改为自适应：
  - 正常：10
  - 429 频率 <20%：降到 5
  - 429 频率 ≥20%：降到 2
- 记录降级事件到日志 + Redis 计数器

##### D. 调用合并优化

- `_classify_and_extract_batch_by_llm()` 已经合并了分类+提取 → 保持
- 新增：当一个 case 有 >5 个材料时，自动走 batch 模式（减少 API 调用次数）
- OCR 结果缓存：相同文件不重复 OCR（Redis 缓存 hash → 结果）

##### E. Worker 并发配置调整

```yaml
# docker-compose.yml - worker
worker:
  environment:
    BAILIAN_OCR_MAX_CONCURRENT: ${BAILIAN_OCR_MAX_CONCURRENT:-10}
    BAILIAN_TEXT_MAX_CONCURRENT: ${BAILIAN_TEXT_MAX_CONCURRENT:-5}
```

**涉及文件**：
- 新建 `services/llm/rate_limiter.py` — 分布式限流器
- 修改 `services/evidence/document_analyzer.py` — 使用限流器
- 修改 `services/evidence/classifier.py` — 使用限流器
- 修改 `services/ocr/bailian_engine.py` — 使用限流器 + 429 重试
- 修改 `worker/evidence_tasks.py` — 降级逻辑
- 修改 `config/settings.py` — 新增限流配置
- 修改 `docker-compose.yml` — 环境变量

#### 1.3 解决分析状态卡死

**问题根因**：
- 任务失败后 status 变为 `failed`，前端无法重新触发分析
- 用户只能删除案件重建

**修复方案**：

##### A. 后端允许 failed 重新分析

```python
# evidence.py — 当前
if case.status not in ("catalog_ready", "analysis_done"):
    raise HTTPException(400, f"Cannot analyze case with status: {case.status}. Need catalog ready.")

# 修改为
ALLOWED_ANALYSIS_STATUSES = ("catalog_ready", "analysis_done", "failed")
if case.status not in ALLOWED_ANALYSIS_STATUSES:
    raise HTTPException(400, f"Cannot analyze case with status: {case.status}. Need catalog ready or retry failed.")
```

当 `status == "failed"` 时：
- 清理旧的分析结果（保留 catalog 数据）
- 删除旧的 analysis steps
- 重新调度 Celery 任务

##### B. 前端显示重试按钮

- `EvidencePage.vue`：当 `case.status === "failed"` 时，第三步按钮文字改为"重新分析"
- 显示失败原因（`case.error_message`）
- 一键重试，无需删案件重建

**涉及文件**：
- `api/routes/evidence.py` — 状态检查放宽 + failed 清理逻辑
- `static/src/views/EvidencePage.vue` — 重新分析 UI

#### 2.1 原告被告身份信息提取增强

**问题根因**：
- 身份证 OCR 文本格式多样，LLM 提取时字段混错
- 多个原告时名字和身份证对不上
- 被告医院名称提取不完整（简写为"XX医院"）

**修复方案**：

##### A. 增强 LLM 提取 prompt（document_analyzer.py）

- 身份证正反面信息合并时，增加"姓名一致性校验"指令
- 明确要求：身份证号必须是 18 位纯数字，不允许带空格/X 以外的字母
- 被告医院名称：要求提取全称，不得省略"市/州/县"等行政区划词
- 新增指令：如果身份证正面和反面 OCR 文本中的姓名不一致，标注 `name_conflict: true`

##### B. 后验证逻辑（document_analyzer.py — 新增 `_validate_extracted_data()`）

```python
def _validate_extracted_data(data: dict, case_type: str) -> dict:

    issues = []

    # 1. 身份证号格式校验：18位，前17位数字，末位数字或X
    for plaintiff in data.get("plaintiffs", []):
        id_num = plaintiff.get("id_number", "")
        if id_num and not re.match(r"^\d{17}[\dXx]$", id_num):
            issues.append(f"身份证号格式错误: {id_num}")
            plaintiff["id_number"] = None  # 清除无效值

    # 2. 被告医院名称完整性校验
    defendant_name = data.get("defendant_name", "")
    if defendant_name:
        if len(defender_name) < 4:
            issues.append(f"被告名称疑似不完整: {defendant_name}")
        if not any(kw in defendant_name for kw in ["医院", "卫生院", "诊所", "保健院", "中心"]):
            issues.append(f"被告名称缺少医疗机构关键词: {defendant_name}")

    # 3. 电话号码格式校验
    # 4. 日期格式统一
    # ...

    data["_validation_issues"] = issues
    return data
```

#### 2.2 多家医院被告识别

**问题根因**：
- 存在转院/多家医院就诊时，LLM 提取到多个医院名
- 但被告只有一个，需要识别哪一家是被诉医院
- 当前 `defendant_name` 只有一个字段，无法表达多家医院关系

**修复方案**：

##### A. 提取 prompt 增强

- 明确要求区分"被告医院"和"非被告医院（转院/会诊医院）"
- 指引 LLM 从被告信息材料（identity_defendant 类别）中提取被告医院全称
- 新增 `other_hospitals` 数组字段：
  ```
  other_hospitals: [
    { name: "XX市第一人民医院", role: "转入医院" },
    { name: "XX县卫生院", role: "首诊医院" }
  ]
  ```

##### B. 后处理逻辑（新增 `_resolve_defendant_hospital()`）

优先级：
1. `identity_defendant` 类别的 OCR 文本中提取的医院名 → 最高优先级
2. 被告营业执照/执业许可证上的机构名
3. 所有材料中出现频率最高的医院名
4. 如果仍然无法确定 → 标记为"待确认"并在前端提示用户

##### C. word_generator.py 适配

- 当存在 `other_hospitals` 时，事实与理由中需按时间线提及各家医院
- 格式："患者于 X年X月X日至 X医院就诊（首诊），后于 X年X月X日转入被告 X医院..."

**涉及文件**：
- `services/evidence/document_analyzer.py` — prompt + 后处理

#### 2.4 事实与理由逻辑 + 时间线优化

**问题根因**：
- LLM 生成的段落 prompt 未要求严格按时间线排列
- 诊疗经过重点不明确 — 该详细的不详细，该简略的不简略
- 尸检报告数据未传递给段落生成 prompt
- 日期格式不统一（OCR 结果中可能混用多种格式）

**修复方案**：

##### A. 6 套段落 prompt 重写（template_manager.py）

**通用规则（所有模板共享）**：
- 强制按时间线叙事：入院当天 → 住院期间 → 恶化/抢救 → 死亡/转院/出院
- 每个关键事件必须带具体日期
- 禁止模糊时间代词（如"某日"、"之后"），必须写明实际日期
- 药物名称+剂量+用法必须写明
- 关键检查结果必须逐项列出
- 禁止 LLM 自行编造因果关系（仅陈述客观经过）

**死亡案件（death_adult/death_minor）特殊要求**：
- paragraph_2 必须包含死亡时间、死亡诊断
- 死亡诊断使用原文，不得改写
- 当有尸检报告时，prompt 中注入鉴定结论作为 paragraph_2 的核心参考资料
- 结尾注明"经尸检/司法鉴定，死亡原因为..."

## 大 PDF（500MB+）专项方案

### 处理流程对比

| 环节 | 当前 | 修复后 |
|------|------|--------|
| 上传到 API | `await file.read()` 全量读内存 | 流式读取，内存 = 一个分片（50MB） |
| 上传到 MinIO | `upload_bytes()` 一次传 | 分片上传（Multipart Upload） |
| Worker 下载 | 全量下载到 /tmp | 流式下载（仅一次） |
| PDF 打开 | `fitz.open()` 全量加载 | `fitz.open()` 惰性加载 |
| PDF 分页 OCR | 全量图片保存在 /tmp | 按批 5-10 页，处理完立即清理 |
| OCR 文本 | 全量文本保存在 /tmp | 直接写 MinIO 临时桶 |
| LLM 提取 | 30K 字截断（粗暴） | 按材料分批提取 + 关键字段定向提取 |
| 临时文件 | 任务结束才清理 | 每批页面清理 + 任务结束兜底清理 |

### 内存估算（500MB 扫描型 PDF，200 页）

| 阶段 | 当前峰值 | 修复后峰值 |
|------|---------|-----------|
| 上传 | 500MB（API 容器） | 50MB（一个分片） |
| Worker 下载 | 500MB（/tmp） | 500MB（/tmp，一次写入） |
| PDF 处理 | ~1.5GB（全量页面图片） | ~50MB（当前批 10 页图片） |
| OCR | 取决于并发 | ~50MB |
| **总峰值** | **~2GB**（接近 OOM） | **~600MB**（安全） |

## 风险与注意事项

1. **prompt 修改影响面大**：每次修改 prompt 需要用已有测试数据回归验证
2. **Redis 信号量**：需要确保 Redis 连接稳定，信号量泄漏会锁死系统 → 信号量键设 TTL（60s），过期自动释放
3. **tmpfs 128M**：对于单页高清扫描图可能接近 10MB，10 页批处理约 100MB → 可调整
4. **流式上传**：MinIO Python SDK 的 `fput_object()` 已支持大文件分片，无需手动实现 Multipart
5. **并发测试**：修改后需要在 10 人并发场景下验证
6. **向后兼容**：已分析完的案件不应受 prompt 修改影响
7. **大 PDF 性能**：500MB+ PDF 的 OCR 将非常耗时（200 页 × OCR 约 1-2 小时），需要确保 Celery 超时够长
8. **音频文件**：当前只做存储，语音识别需后续单独开发
9. **OCR 结果缓存**：用 hash 去重相同文件，减少重复 API 调用
